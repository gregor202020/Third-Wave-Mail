1. **SNS idempotency** in `packages/api/src/routes/webhooks-inbound.ts`:

   In `processNotification()`, modify the Bounce case (line ~191) and Complaint case (line ~231) event inserts to use `onConflict`:

   For the Bounce case, change the `db.insertInto('events').values({...}).execute()` to:
   ```typescript
   db.insertInto('events')
     .values({
       event_type: eventType,
       contact_id: message.contact_id,
       campaign_id: message.campaign_id,
       variant_id: message.variant_id,
       message_id: message.id,
       event_time: new Date(),
       metadata: { ... },  // keep existing metadata
     })
     .onConflict((oc) => oc.columns(['message_id', 'event_type']).doNothing())
     .execute()
   ```

   For the Complaint case, apply the same `.onConflict()` pattern to the events insert.

   The contact status UPDATEs are already idempotent (SET status = X WHERE id = Y) — no changes needed there. The campaign counter increments (total_bounces, total_complaints) are also safe to run twice because SNS dedup prevents the second execution from reaching this code path due to the early return from ON CONFLICT DO NOTHING. However, the current code uses `Promise.all()` which means the counter increment runs in parallel with the insert. To be safe: check the result of the insert — if 0 rows affected (conflict), return early and skip the counter increment and status updates.

   Specifically, refactor each case to:
   ```typescript
   case 'Bounce': {
     const bounce = data['bounce'] as Record<string, unknown>;
     const bounceType = bounce?.['bounceType'] as string;
     const isHard = bounceType === 'Permanent';
     const eventType = isHard ? EventType.HARD_BOUNCE : EventType.SOFT_BOUNCE;

     // COMP-01: Idempotent insert — returns 0 rows on duplicate
     const insertResult = await db
       .insertInto('events')
       .values({ event_type: eventType, contact_id: message.contact_id, campaign_id: message.campaign_id, variant_id: message.variant_id, message_id: message.id, event_time: new Date(), metadata: { bounce_type: bounceType, diagnostic: (bounce?.['bouncedRecipients'] as any)?.[0]?.diagnosticCode, sub_type: bounce?.['bounceSubType'] } })
       .onConflict((oc) => oc.columns(['message_id', 'event_type']).doNothing())
       .execute();

     // If duplicate (0 rows inserted), skip all side effects
     if (!insertResult || (Array.isArray(insertResult) && insertResult.length === 0) || (insertResult as any).numInsertedOrUpdatedRows === 0n) {
       break;
     }

     // Side effects only on first processing
     const bounceOps: Promise<unknown>[] = [
       db.updateTable('messages').set({ status: MessageStatus.BOUNCED }).where('id', '=', message.id).execute(),
       db.updateTable('campaigns').set((eb: any) => ({ total_bounces: eb('total_bounces', '+', 1) })).where('id', '=', message.campaign_id).execute(),
     ];
     if (isHard) {
       bounceOps.push(db.updateTable('contacts').set({ status: ContactStatus.BOUNCED }).where('id', '=', message.contact_id).execute());
     }
     await Promise.all(bounceOps);
     break;
   }
   ```

   Apply the same pattern to the Complaint case: insert with onConflict, check numInsertedOrUpdatedRows, skip side effects on duplicate.

   NOTE: Kysely's `.execute()` on insert returns an `InsertResult` with `numInsertedOrUpdatedRows`. Check `insertResult.numInsertedOrUpdatedRows === 0n` (BigInt zero) to detect the conflict/no-op case.
